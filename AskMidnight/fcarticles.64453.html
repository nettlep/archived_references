Graphics Interface Layer
Question submitted by (22 January 2000)
 
  I like to write an graphics interface layer that can use OpenGL and Dx internally. I read your solution in the FOK column but a still confused about the implementations. I can think of 2 solns.

1. Create an abstract base graphics class and derive classes that use OpenGL or Dx implementations. I don't like this because it requires virtual functions and as a graphics API, speed is a concern.

2. Create two seperate auxillary class that implements OpenGL and Dx8 specifically with same interfaces(implemented in a dll) and graphics interface calls the auxillary class. This is a maintanence nightmare and it's quite troublesome too.

When I decide to add a texture object class that can be implemented by using a Dx surface (dx8) or a series of bytes (OpenGL), both methods does not allow it easily without some ugly friends access (and other nightmarish practices).

I would like to make function calls like this

// use the object to set the texture in graphics object
Texture_Object.SetTexture(Graphics_Interface_Object);

and

// use the graphics layer object to set a texture
Graphcs_Interface_Object.SetTexture(Texture_Object);

What are your suggestions? 		 
 

 
  Personally, I would go with #1, but make sure to give your API a high-level interface.

The high level interface will reduce the number of calls placed into it. For example, you won't be calling it per polygon, you'll be calling it per mesh. If you're worried about the calls you do place through the virtual mechanism, don't worry about it. The difference is probably only about 0.001 frames per second, which is a small price to pay for maintainability and cleaner design.

The places where you'll loose a lot of performance from the virtual mechanism is in your low-level APIs, (like a matrix/vector class, for example.)

I think you're running into problems with your Texture_ObjectSetTexture() example because you're not working at a high-enough level. Try placing your texture information in the object data (or class) when you pass it to the rendering DLL.

I did it the same way Paul proposed and things worked fine for me. Just use abstract base-classes for all classes that are based directly on the API and it works fine (with having thought about the design previously). For example the Nebula Device 3d-engine does it this way, just have a look at http://www.radonlabs.de .

As to your texture-example, things would look like this (in my engine) :

vsTexture (abstract base class)
1) vsOpenGLTexture
2) vsD3D7Texture
3) vsD3D8Texture

and the base-class has functions like Init(char*, vsTextureDescriptor*), Bind() and release() which are the ones to be used. One problem can be the creation of new objects based directly on an API, so in my engine all the basic object-types (there aren't many, things like dynamic and static world-nodes, single textures, and a few more) are created directly using the API-specific base-class, so the wrapping works fine for me.

The speed problem can occur when you've got a too big overhead within your engine, but that has something to do with the overall-design and not with 10 derived classes. I heard the guys at Ensemble Studios rewrote their code of AOE2 to use virtual functions and derived functionality and had a speed loss of about 4%. And I guess they didn't use it in about 10 classes, but many more, so speed won't be an issue. The maintainability and nice code-design is really worth this little speed-loss.

I've worked on a 3D engine for the past year or so, and over this time I have gained some experience in making a large OOP project. I adopted some design patterns which I developed on my own, to be consistent. Basically, my engine is module-based. Its design can be viewed on several levels:

System APIs
Supports
Core Engine
Extensions
Tools

The system APIs shoudl already exist. The rest is part of the engine.

The core engine contains modules consisting of virtual interfaces. This is an engine layer that defines the functionality of the core groups, such as graphics, rendering, window management, gui, world management, etc. Thinking about common-denominator interfaces and what is based on what is the most xchallenging part, imho.
Supports are modules that derive from modules in the core engine, and implement them on a particular API set. The usaer/programmer can choose which set of supports to create, and once they are created, the interface is already known.
Extensions are new modules that do not supply core functionality, but instead house pointers to core engine objects. They use these pointers to controls the engine objects, but perform different things with them.

Normally, only the Supports should be API-dependent.

In my view, textures cameras, etc. are to be treated as passive objects. That is, they only affect themselves. Therefore, the graphics system should set the texture, etc. This can also be evidenced by the fact that one texture can be set to several graphics systems, perhaps.

Anyway, those are my 29 cents.

I've gotten around the virtual-functions-are-too-slow-but-I-need-that-flexability problem by using function pointers.

Take this simple display "class" as an example 




typedef

  struct

{

  void (*InitDisplay) (void);

  void (*GetProperty) (DisplayProperty *Property);

  void (*SetProperty) (DisplayProperty *Property);

  /* All sorts of rendering functions */

  void (*SwapBuffers) (void);

}

  iDisplay;


Each DLL has the following function that fills in the fields of the iDisplay record (I'm assuming you place rendering code for each API into separate DLL's) 



void dllDisplayInit(iDisplay *Display);


and the main application has the following code 



iDisplay gDisplay;



int

main

  (int argc, char **argv)

{

  LoadRenderApi();

  dllDisplayInit(&gDisplay);



  gDisplay.InitDisplay();

  /* ... */



  return 0;

}



For something like a low level display class this works beautifully.

Well, I have to go with Mark Friedenbach. My engine has a similar method for the main rendering system, using function pointers that are filled in when the DLL is loaded (from a function in the DLL that gets a pointer to the RenderSystem class and just assigns all the pointers), this also saves the time of having to create a new object for the system and deleting it after. Speed-wise, the most recent method seems to work the best without bogging it down with unnecisary calls.

There is no gain you get with function pointers over virtual table. It's actually the opposite, because if you check the implementation for v-table in VC++, you see, that it's not struct of pointers, but list of jmp [pointer] commands, which is more efficient on modern computers than function pointers (why? check Pentium specs).

This also lets you have virtual base class (no, i'm not talking about virtual methods, but virtual inheritance, eg. class A: virtual public B {};)

So just stop tweaking things, because compiler does this better. Also if you fix your implementation of virtual function calls to be usage of function pointers, it can't be changed (read, optimized).

I have had this kind of abstract interface design in my engine and it works very efficiently, because of such high-level interface Paul suggested. Alot of calls are performed behind that interface to non-virtual functions or inlined functions, so when you design abstract class and derive implementation from it, you definately don't need to have all your calls as virtual.

The advantages of virtual functions and why you should use them:

1) People mroe easily understand your code, and compilers can optimize for it better
2) They are well thought-out and fast.

You see, virtual tables are stored once in memory and take up less space, and a,lso can wind up in the cache better since they are only once. The virtual table pointer is at the beginning of the class so of course it winds up in the cache. It's fast and easy. If you want to call functions, there is no faster way than dynamic linking using function pointers - -except using static linking, because it replaces the addresses in-place so they're constants.

In short, use virtual functions. They're a well-known mechanism and they're fast.
