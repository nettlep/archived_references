  I sent this one off to Tim Sweeney, and here's his reply:

"The Unreal file format is an indexed serialization stream of a bunch of interrelated objects. Parsing it without the code is pretty much hopeless -- it's a different concept than a .bmp file or a .wav file. See http://unreal.epicgames.com/packages.htm for more info."

   Use .t3d files instead
I use UnrealED to make levels for my 3D engine, but I don't use .unr files.

The way I do it is this: make your level as normal in UnrealEd, saving as .unr for working purposes. When you want to export to your game, create a cubic brush bigger than the level surrounding it, then de-intersect it. This creates a brush with all the polygons in your level, nicely split up into convex shapes. Choose Export from the Brush menu and save it as a .t3d file. This is ASCII and fairly easy to parse. You'll need to export all the textures you use though. (I'm not sure if there is a size limit on brushes, I've only made small levels so far).

This only gives you geometry, however. To put actors in a level for monsters and stuff, go back to UnrealEd, place monsters as you would if you were making an Unreal level. Now go File->Export and this will export the whole level and the actors in it - the reason I don't use this is it keeps things in a CSG format for Unreal levels, ie stores rooms as subtractive cubes instead of a simple poly list. However, there is an actor section you can parse easily enough to determine the locations of actors and their class names. Pass this file along with the exported brush to a utility program to convert the files into a format suitable for your engine, and it can combine the two into one level. Class names will come out as the name of the class you put down in UnrealED - if you create your own (empty) classes derived from, say, a SkaarjWarrior, it will record the name of your derived class in the file, so you can use your own monsters/weapons etc.

NB:
.t3d files seem to miss out attributes of actors if they are at their default values: eg if an actor is at location (0,0,0) then the location is not recorded in the file.

The axes seem to go x left, y forward, and z up, if you see what I mean.

UnrealEd doesn't seem to like exporting to a directory with spaces in it's name. I just export to the root and then move them myself.

Hope this helps!

   Re: Use .t3d files instead
I hit the same snag myself, and I was really stuck until I found out that in a .t3d file they were stored as vectors instead of 'useful' :) u v co-ords. What I do is, find the first point on a poly (p0), set its uv values to 0. Then for subsequent points, subtract the position of the first point from them to get the vector going from p0 to it, dot-product this with the texture's u-vector, divide the result by the u-vector's length, then divide by the texture's width. This gives you the u-coordinate for that point. Do the same again using the v-vector and you get the v-coord. This means you need to find the dimensions of the relevant texture, so make sure your converter program has access to the textures of your game program.

This does however mean the same textures on adjacent polies won't match up quite properly, I think you could use the origin as p0 instead of the first point on the poly. This would give you high co-ords though. This also doesn't take into account the scaling of the texture, at the moment I'm only interested in test levels though :).

I don't think I explained that very well, but if you go through the maths I think it should be ok... :)

"This does however mean the same textures on adjacent polies won't match up quite properly"

I don't have any experience with Unreal map files, but I'd guess that they use are using planar texture mapping which is why they are storing vectors instead of coordinates.

The idea behind planar texture mapping that you have imaginary planes tiled with your texture texture and you then project the vertices of your polygons onto these planes. If you project neighboring polygons onto the same texture plane then you textures will match up quite nicely at the edges (although typically you will get a little distortion because of the projection).

You can read an explaination of how it's done in Quake 2 here. The might be doing something similar in Unreal.
