Software Protection
Question submitted by (04 April 2001)
 
  Do you know any good algorithms/techniques for generating unique CD serial numbers to help identify legitimate customers at game install time, game play time, and to allow access to online games and forums?  
 

 
  This is a little off-topic, but as a shareware author, I happen to have an interest in protection against piracy & cracking (i.e. 'reversing').

Let me start off by saying that I don't know anything about the cost or overhead involved in burning a unique serial number onto each CD. I can say, however, that this will do you very little (if any) good at all. Crackers have some very sophisticated tools (SoftIce is the favorite) and they are more on top of this stuff than you or I. I guarantee you that they are also more on top of it than those companies you can purchase your protection from (i.e. hardware dongles or software-based encryption/protection schemes.)

So, let's talk about protection... This is based on stuff I figured out on my own, and while talking to people on efnet's #cracking4newbies IRC channel. If you go there, they'll help you by giving you information on how to best protect your program (they're actually a very friendly group of people.) They'll also crack your program and distribute it. So use caution if you don't want this to happen. That is, after all, what they do.

When a cracker gets his hands on a program with a key, he has two options. (1) He can crack the program so that it no longer requires a key, or (2) he can reverse-engineer the code to figure out how the key works and use that information to create a 'keygen'.

There is a third option called carding. Carding is using a stolen (real) credit card to buy a key and then distributing that key publicly. I don't condone the act of cracking, but I can at least appreciate the talent that is required by cracking some of the more difficult programs out there. However, I find carding rather distasteful in every respect (and, as I understand it, the cracking scene agrees with me on this.) It's also credit card fraud, and a federal offense. Reversing, however, is not necessarily illegal (at least, not until you distribute the crack.) You can help prevent carding (or at least, slow it down) by having a server-side serial number (key) check. If you find any carded keys being publicly distributed, simply shut them off on the server. This could open up two other cans of worms... this might make your server prone to hackers (to get keygen information) or you might find that each time you shut down a key on the server, they card a new key.

Can a program be made to be completely impossible to crack? No, not if you consider what a cracker is willing to do (even hacking into server for server-checks.) Even though I've been told (by crackers) that a program can be made uncrackable, I personally don't think so. Instead, my goal has always been to make the protection more effort to crack than a cracker is willing to expend. And they're willing to expend a lot of energy for a $5 program. It has nothing to do with money; it is all about the challenge. It's important that you understand that when you protect your program, you are challenging a cracker to crack it.

In order to protect your program (and you will need to do this, because the toughest key is useless if the program itself is easy to crack), you'll need to have a small idea of how a program is cracked, and what tools they have at their disposal.

There are some extremely sophisticated disassembles on the market. These tools are often used by driver developers and even by large companies. For example, IDA (Interactive DisAssembler) is one of the better tools. They boast that it's used by government agencies to validate commercial software, by open-source activists to support undocumented architectures, by car tuners to make modified chips for cars, etc. I downloaded the demo of this, and was amazed at what it did with my program. To the trained eye, this is almost as good as having the source code.

So, how do you protect yourself against a cracker, when they've got the source to your program? I don't know. I don't think you can. But I think you can make it more effort than it's worth. At least, that's what I'm hoping for from my protection.

When a cracker wants to crack a program they know that more often than not, the protection will simply result in a conditional jump or two. You might have an RSA encrypted key, but if your result is to simply check to see if that key is valid, you're going to end up in a conditional jump someplace after all the decryption code has run. If they comment out that jump instruction, or switch the condition, then your protection has been foiled.

My approach was a bit different. My approach was to create a key that is required for the program to run. Not logically, with conditions, but by actually using the key in the code itself. For example, there might be a math formula stored in the key. All keys have different values, but after the formula is run, the result is always the same across all keys (i.e. 10-1=9 and 7+2=9). I might use this value as a stepping value through a loop. If I packed my key with a lot of formulas, and used them in various places in the code, then the key is an integral part of the program. But will that stop a cracker? Nope. They just find all the places where I use the key, and crack them individually. This time they're not swapping out conditional jumps they're replacing the key calculation code with a few instructions that simply place the result from a formula into a register. And they can easily find all the places where I use the key, by using the disassembling tools and searching the ASM source.

But I think this is a good start, so I decided to enhance it a little. My goal was to place so many key checks in my program that they couldn't possibly crack them all. At the same time, I didn't want to pollute my program's source with thousands of key checks. As a solution, I decided to use a few simple #defines.

After a few well designed #define statements (so I was sure everything was inlined) I was able to proliferate my key checks so much, that my executable grew by 300K. My key checks are 33% of my program's size. So, as an example:

    #define sizeof(a)  (keycheck_calculation + sizeof(a))


If that particular key check calculation were supposed to result in zero, the program would run fine. However, if an invalid key was used, the result drastically affects the way the program operates, and probably makes it crash.

There's still more work to do. You have to secure all aspects of the program as tightly as possible, because a cracker will find the weakest link. For example, my program requires a registration code to run. The trial version of the software simply disables the save feature. If I were able to allow people to register the trial version by simply entering a key, then the cracker could simply crack the trial version to enable the save.

Instead I would need to have two separate versions of the program (trial/registered.) The trial version would have the save code conditionally compiled out, so there's nothing to crack. Or is there? Couldn't the cracker compare the differences between the trial version and the registered version and simply add the code to the trial version? Yes, they could. Fortunately, with my key checks proliferated throughout my registered version so thoroughly, finding this extra code would be a nightmare. Not impossible, but very difficult, especially if the needed code from the registered version was littered with key checks.

This was about as much effort as I was willing to put into it (at this point, it was about an evening's worth of work.) Since it was just an evening's worth of work, if I found that my program was cracked, I would simply change the protection in the next version. After all, I was writing a shareware program, not a protection library.

One thing I haven't covered is the actual key generation. Ideally, you want your keys to be non-reversible, even if the cracker has the source code to see how they work, they don't have enough information to actually create a valid key (much like how many popular encryption algorithms work today.) I won't go into how mine works. Besides, I'm not an encryption expert by any stretch of the imagination. My goal was to make it (what I thought of as) un-reversible, and hide the rest in obscurity. :)

There are a lot of other aspects that I haven't covered (like checksums in your program to make sure it doesn't get patched, debugger-detection, etc.) I didn't cover them because most of them are useless. At best, they'll slow the hacker down and waste a few minutes their time. More importantly, I didn't cover so many aspects because I'm simply not even aware of them. I'm not an expert. If you really want serious help on this topic, go find an expert cracker that's willing to help you.

   Is all this really necessary?
As I type this, my shareware program got the 5th registration of the day. So far, for this month, that's an average of one registration every 8 hours.

I can guarantee you that this wouldn't be the case without this protection. At one point, there was a keygen available for my program that didn't work (it simply generated keys that survived the checksum test with a checksum of 0.) The keygen, however, did not survive the crucible to follow. :) The software would act very flaky with an invalid key. I even got a bunch of emails from people complaining about these problems. [shrug]

The software automatically checks for updates of itself. In order to do so, it must query the server for a text file. When you send a query to a web server, you include the browser information. In my case, the browser was shown as "3D Mail Effects Version Checker (S)" or "3D Mail Effects Version Checker (R)" -- The S and R mean 'shareware' or 'registered'.

Well, once that keygen was released, I noticed a LOT of people running the registered version. I knew about the keygen, and I knew that it wasn't working, because I could watch my HTML logs, see somebody appear running the shareware version, then the registered version, then try a bunch of times with the registered version, and finally, they would be running the shareware version again. This was a very common occurance. Over 80% of my logs were full of this kind of activity.

I also watched my sales drop. When I put out the next revision to the software, it no longer allowed keys with checksums of 0, even if the checksum was correct. This defeated that particular keygen. From that point on, the suspicious activity went away and my sales picked back up.

I almost felt like Frankenstein held up in a castle with the torch-bearing city dwellers outside just waiting for the door to open so they could flood in. Obviously, I did the prudent thing and tightened the locks. :)

I thought of a couple of ways to make things "impossible" -
1) Using weird math functions that has no real meaning (or even ones like counting how many times a digit is found in a number). "WhatIs" style of function should really help.

BOOL IsRegistered();
Patching such function so it will always return true, won't take more than 10 seconds.

Software protection is very difficult to get right and depends largely on who will use your program. It sounds like Midnight's program has very wide appeal but not use by companies...

I small company I worked for used quite a basic check for registration but was tied to the machine. This was okay for us but meant that users had to ask us for a new key if they changed their machine. This was required to ensure that one copy was not simply installed on a whole department full of machines... (quite a possiblility in this case).

There is an alternative to protection and that is to try a different business model (so that protection is not required). The sucess of this approach usually depends on the type of software though.... The extreme example of a different model is Open Source. Essentially, your program is free but users pay for added value. I have had people talk at me about this several times and I am not convinced. It might work for large programs but $5 utilities rarely enter into it. One related option is to get a company / user who wants a new feature to pay for it, on the condition that the final product is released to all. I have heard that several people have done okay on this.

The one I most like though relies on honesty and the product having company appeal- the 'free for personal use' licence. Companies are more likely to buy legal software and are more likely to be able to have bigger funds (eg: they *wanted* to pay 4000 for software which we thought worth no more than 500).

If you are writing software with a view to making money then pick your product carefully. I personally think that writing the software because you wanted to and then trying to make money from it is nigh on impossible - my stuff is freeware....

Just my 2p worth.

as soon as a cracking group gets hold of a leaked serial/key, you can
be sure they'll work something out.

RSA will be factored if it's small enough ("small" being a number that
constantly increases, due to faster CPUs, and soon probably also distributed
methods).

IDA might be one of the only applications I've seen that has a very
good scheme. Watermarks all over, so you can be pretty sure that a
person who bought it will NOT be leaking it. But 4.04 got to the
cracking scene after all, the "gary colton" carded release.

>"Using weird math functions that has no real meaning".
Why, if they have no meaning? Sure, they will work as obfuscation,
making it a bit harder to find the spot where you need to apply the
sting (which is what cracking is all about 90% of the time, finding
"that magic location"). But these days, it will take a LOT to fool
the crackers you are up against. There are *very* sophisticated tools
out there for crackers, by crackers.

The icedump for softice project deserves a look - it's amazing.
http://icedump.tsx.org

up a lot before thinking up a method. Fravia's not-frozen mirror at
http://tsehp.cjb.net shows how many crackers think, and is an invaluable
resource, for crackers as well as protectionists. Be sure to read
some of the older stuff as well.

If you want to keep crackers away, you have to learn to think as a
cracker. It's not enough to have something that looks neat in visual
studio - you have to get down to the nitty-gritty asm stuff - have
a look at your code in a dissasembler and a asm-level debugger.

The most important thing? NEVER, *EVER* buy ready-made protection.

always be a generic unpacker around. The harder the challenge the more
skilled crackers (who deserver to be called reversers) will join the fray.

Face it, if a program requires a static key, that's it. Anyone can pirate it. Just down a program and the related key.

The only "uncrackable" software is software that connects to an uncrackable source (e.g. server on the internet) for vital info every time, or once every day, or something.    Re: Software Protection - proposed solution
After reading all the great comments and thinking about this issue some more, it seems to me that the main problem is that people don't look at software as property. Many MANY people who'd never steal a physical possession or even shoplift some candy will steal software in abundance with out a concideration. Doesn't make them bad, but it does mean we can't be naive and not protect our software.

With that in mind, taking comments from everyone here and some ideas by Microsoft, I thought I'd propose a solution (Not bullet proof, but very limiting) that might work and see what people think.

Essentially the idea lies in generating the keys based on some basic system info, proc speed/brand, mother-board info, ram info (speed, size), OS serial number (questionable), and Serial number for the program. All of this would generate the number and then the user would either call or go online and register this number with the company. The company could then decode this and fill in the information in their database (Nothing would be collected that would conflict with privacy issues). The server would then generate a confirmation number which would be sent back to the user via the web or phone.

This number would be valid as long as certain pieces of hardware wasn't changed. As soon as that happened then the number would be voided and the program would attempt to autogenerate/verify another number with the company server through the web and provide the user a number to call if that doesn't work. They'd check that number, decode the product serial number, compare that to a database of sold product serial numbers, and void the old number.

The verification code in the program would be implemented as Midnight suggested, obfustication being the goal, not total prevention, so that by the time a crack came out your product would hopefully go through one sales-cycle uncorrupted. Then a new version or patch could be released to make them go through it all over again.

Some issues would be if someone steals a valid key, how would you prevent them from replacing a valid users? This would be done by requiring the user to create a user name and password when first installing the program, which would be how the company would verify your legitimacy to that serial number.

Lastly, just to be a real bitch about things, the program would auto-verify it's self every time you logged onto the web. This would be done in the background and would just be an extra measure to make it annoying to crack.

One other idea, though I don't like it, is to combine the obfustication method with actually making the program break your system if you try to run a cracked version and providing very clear forwarning. A popular FTP client does this, but I can't recall which. Makes it REALLY annoying to crack it. Again, this last bit I don't really like. The goal is just to make it so that it takes a long time to crack and therefore you get a chance to maximize sales.

Ideas? comments? Seem valid? Holes? I know it'd take a bit of work to implement, but then again, you'd be able to reuse it on all your products as well as long as you changed your obfustication.

   Use public-key crypto (was: Software Protection)
The question of software piracy boils down to the question of crytography.

Unfortunately, the fact is that for Internet-less games, the computer is a closed system with no trusted source and, therefor, will be cracked. Whether it's "faking" a CD-ROM drive, or cracking the code itself, or discovering the private key built in to the software, whatever scheme you come up with will be reversed. And fast.

With the Internet at our disposal, it's much easier: you can use public key cryptography to force the user to connect to the correct server, and thus you have a trusted source that can generate some kind of valid start sequence/packet. If you're really paranoid, use an encrypted connection to hide the authentication sequence (and thus will be harder to spoof at the TCP/IP level). Then your server can do things like

(a) Look at the private key of the game, generated at install time (or maybe the user's credit card number? Or username/password?)

(b) Study statistics of usage information, i.e., why is user 123 logged on 75 times from all over the globe?

(c) Get demographics; how long do people play in one sitting? How far do they get? Did anybody discover the secret treasure room? You may have ethical issues to consider... :)

(d) Query the user's setup (what is the checksum of the binary they are currently using?) Again, ethics here...

Some closing notes:

About 50% of sales occur in the first 2 months. Delay crackers for that long, and you can count yourself lucky.

I'm a Linux/Open Source nut, but believe in commercial software. If you use public key crypto to prevent piracy, ship the source code of your game engine along with the (copyrighted, protected) game data. Then maybe some kid will learn from it and build me a holosuite 20 years from now.

Hey guys,

I just briefly scanned through this page so this issue may have allready beed discussed and I missed it,
but anyway, here`s what I`d do (I don`t know if it would be any good):

In your program you have one or more vital procedures, like main loop etc...
Solution is you encrypt the code of these procedures with some strong symmetric process (you can make your own).

Let`s imagine your normally compiled and linked program.
=====
When you write and link the program all the code is converted into numbers and packed into code area. After this block constants come and then static variables` block. And this is EXE file (plus the header).
After you run the program, EXE is just read into memory as a block of numbers at offset 400000h. Memory pages that hold code are marked as read-only and you cannot write into them not causein a page fault. Same is with constants. On the other hand static variables block is marked as read-write and you can write into it whenever you want.

At this point your vital functions exist unencrypted at some specific offsets in the memory. When they are called, they are executed.

The idea is to take the hex numbers of those valuable functions (areas of code) and encrypt them hard. If they are encrypted, CPU understands those numbers as totally different commands as they originally are. When user starts the program, he needs to enter the password (or something like that), then the password may be converted into a hash and with this hash you then operate on the encrypted blocks of code. If the hash was correct, you`ll come up with the original code. If it wasn`t correct, you`ll get just rubbish bytes. When CPU then tries to execute those bytes, if they`ve been unencrypted correctly, everything is OK, if not, program will crash to desktop.


How to do it?
=====
Under DOS you could do this:
Compile the program into exe. Find the offsets of your procedures and encrypt those areas of exe file. Then when file is executed, it itself tries to decrypt those areas.

Under Windows, code area is read only and the upper process would cause a page fault.
So, to mangle with the code areas, you need to take them out.

I`d do this:
Delete all your valuable functions out of the source code.
Write a program that will dynamically allocate blocks of memory for your procedures, and at places where you need to call these procedures, call these allocated blocks instead.
If these procedures need any external references (global variables or other functions from allover the program), pass those offsets on stack.
Compile and link.

Take the source code of your vital functions and assemble it manually.
If you need any of the global variables or other functions, you cannot address them directly since the offsets change at every link time (if you changed the code). Instead, program will pass these offsets on stack so expect them there.
You must make all calls non-relative (absolute, so the are not offset dependent).

Now, compile this offset-independent code. You get a bunch of hex codes.
Write them down into a small file and encrypt it.

When main program executes, it reads this small file with encrypted code, tries to decrypt it and executes it.


What are cracker`s options?
=====
He can extract the decryption procedure from the main exe. Then:

a) Steal some valid password and corresponding encrypted code files (you cannot fight against this)
b) Run a brute force attack on encrypted code files
