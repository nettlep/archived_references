Performance Questions
Question submitted by (04 August 2000)
 
  I have read in many tutorials that aligning memory correctly can give a large performance boost, how should your vertices be aligned for the best performance, and how do you do this? Also I program Opengl and was wondering whether sorting all the data so that unnesecary texture changes are avoided will give a large performance increase?  
 

 
  Most of your C/C++ compilers default to 8-byte memory alignment, which is fine. Don't sweat this detail.

Sorting textures, on the other hand, is something you'll want to spend some time on. The reason this is an issue, is because the 3D cards don't actually render from their texture memory. They copy the current texture into a temporary buffer that has faster access during rendering. This is why a texture change is so expensive - they have to copy the texture to this temporary buffer. The buffer can only hold so much, which is why so many cards were bound to 256x256 textures for a long time.

This is less of an issue with newer cards, but it's still something to be concerned about.

Other state changes can also cause a performance hit, but again, this is less of an issue with newer cards.

   Re: Ask MidNight
Texture state changes are expensive because they create bubbles in the hardware rendering pipeline. These bubbles are created because their is a dependency between commands, sent from the driver to the hardware, and the information stored about a texture in certain hardware registers.

When a texture state change occurs the hardware needs to finish rendering geometry that relies on the previous texture before the new texture information can be programmed into the registers.

Copying texel data into a temporary buffer does not seem practical. How big should this temporary buffer be? It would be a waste that we copy an entire texture if only 1/8 of it was ever used. Hardware today can process several textures units per pipe so that multitexturing can be done in a single pass. Does that mean the hardware needs several temporary buffers?

What seems more practical is the hardware logic to fetch data quickly from local and nonlocal video memory into a local FIFO as needed. System memory texel fetches would be much slower.

   Re: Ask MidNight
I am not saying his point is totally incorrect, but it was not made clear why the texture copy to a local buffer is expensive. The actual copy should not be very expensive.

I could imagine yesterday's graphics cards used temporary caches to store texture information, since the textures often times sat in system memory. It would be an optimization to prefetch the texture to some local storage. The rules were different back then. Consoles are a different story. They have their own rules. :)

Today, we have AGP memory and alot more local video memory that graphics hardware has access to. So, there isn't much need for a temporary buffer that contains a copy of a texture. Don't get me wrong, I am not saying that today's hardware doesn't use buffering techniques.

However, the reason hasn't changed for why texture state changes are expensive. It is also the same reason that locking vertex buffers for writing can be very expensive if done in the wrong place, like in the middle of a render loop. It is the bubbles created in the pipeline due to the state changes or the locks that make these actions expensive.

   Re: Ask MidNight
Midnight is right in some ways. All modern cards have a texture cache (often both l1 and l2 caches), and they must be copied to. Often l2 storage can store most of a texture. Often with re-accessing the same texture can however save bandwidth because of this. Video chips don't have the video memory bandwidth to read 4 (8 in the case of trilinear) 32bit texture samples every cycle. Often more in the case of multi-pipeline multi-texture (look at the geforce2, without a texture cache, it would croak).
