Good Programming Practice
Question submitted by (06 February 2002)
 
  When writing a program for stabililty, what should programmers look out for? (Aside from using your memory manager!) :) Any tips you can share for making sure our code is more stable than say - Adobe Premiere? ;) Techniques/things to look out for etc...  
 

 
  This is a great question, one I'm sure will be flooded with reader's comments to follow. However, to properly answer this question would require enough content to fill a number of books.

Fortunately, those books have already been written by people much better at this stuff than I am. Three books come immediately to mind: Code Complete (by Steve McConnell, Microsoft Press), Writing Solid Code (by Steve Maguire, Microsoft Press) and Effective C++ (by Scott Meyers, Addison Wesley).

You might be asking yourself, "Books?! Is that all the wise-and-all-knowing MidNight can offer us? A list of books?!" Of course not. Rather than deluge you with every tip I can think of off of the top of my head, how about three generalized tips? Possibly the most important tips of them all.

1. Know your language (assumes C++)

Did you know that a proper implementation of operator new requires an infinite loop? Or how about properly copying a list of objects includes the requirement of in-place construction with operator new and destruction of elements of the list?

How can you expect to write stable and robust code if you don't know the language you're writing it in?

There are a lot of hidden details in the C++ language. I learned this first-hand when I created my personal template library (lists, arrays, hashes, reservoirs, strings, etc.) Sure, you can make a list class, but can you make a proper list class? I probably only know a half-dozen people that can.

C++ is a horribly complex beast with a very misleading "simpler" side. If you only learn the simpler side of the language, you're missing out on the true beauty of the language. And wrapped up in all that complexity is a lot of power. Take command of that power and learn the language you're writing your code in. It'll save a lot of headaches.

Of the three books mentioned above, Effective C++ is the best book for this task. The book is organized in a series of "Items" (similar to the tips you're looking for) that can be read out of order so you can jump to the ones you're most interested in. However, I recommend going through the book very thoroughly. As a matter of fact, it's the only technical book I've actually read cover-to-cover without picking and choosing the topics I wanted to read. And I can honestly say I'm a much better programmer because of it – it had a very severe impact on how I write code and taught me a lot about the language. More importantly, it spiked my interest in learning the language to a greater degree.

It also won't hurt to spend $18 and get an electronic copy of the ISO specification for C++ -- I find my copy invaluable at times. Though it may be insufferably dry at times, it can answer any non-compiler-implementation dependent question you have about the language.

2. Plan your coding

So you've got this great idea for a cool program. You open up your compiler and start coding away. Two weeks down the road, you realize it would be great to add this one feature, except doing so would require a huge re-write of many sections of the code, because the program wasn't organized in a way to allow you to have access to the data where you need it. Too bad, you shoulda planned ahead!

Unless you are writing a throw-away program, spend a little time up front and plan it out. At the very least, map out the primary and secondary components; how they relate and what each component is responsible for. Doing this simple task will help you see how everything fits together and where some holes might be. I find that this also helps with ideas for cool new features to add. In the end, it tells you where to put all of your code, development goes faster, and the end product tends to be much more stable.

If you're going to spend a few weeks on a program and don't want to spend the time for a proper design, at least spend a few hours up front doing this. You won't regret it.

3. Be patient, practice

No matter how experienced you are, every you ever write is practice for the next program. The bigger your mistakes are, the more you'll learn.

If you're not making mistakes, you're not gaining any real-world experience.

I'm in the beginning stages of a really big project ... fortunately it's a learning / hobby project, so I can take all the time in the world I need in order to do things The Right Way. :)

Something that has already saved my bacon a few times, and as things get truly complex will be a lifesaver (I hope!), is a good unit test suite with accompanying harness.

I'm in a unix environment and using the standard tools, and it is very very cool to be able to:
make
make test

And see that everything still works after a series of big changes. For interactive UI stuff / things that need a human to see if they 'look right' or not, I have a simple 'wait time' for an operator to press ESC (test failed) or return (test passed). If the time goes by, the test is left as undetermined.

Super handy :) but a bit of a time suck at first. I'm getting pretty disciplined now about writing a test simultaneously with a new feature, and it's getting a lot easier / more efficient.

I love this subject! Here are some of my "10 commandments", many of them probably stolen from Code Complete.

SLOW DOWN
If you take twice as long to write each routine/object, you'll probably finish your project in 1/2 the time. It's the little bugs that will kill you in the end.

ASSERT EVERYTHING
It may sound crazy, but I use Assert()s to validate almost every parameter. This goes hand in hand with the SLOW DOWN mentality. Asserts are a good way of keeping you "honest" with your logic. Hey... they don't cost anything other than typing time.

KEEP IT SIMPLE
If your solution to a problem seems like it is getting too complex, then it is probably wrong. Rethink and attack the problem again. You may need to learn from your mistakes to find a graceful solution.

MAINTAINABILITY OVER PERFORMANCE
Most of the time clarity and maintainability are FAR more important than performance. Don't play cute coding tricks to shave 0.002% off the loop time of your program. You're CPU is probably idle waiting for the video card anyway!

FIX BUGS IMMEDIATELY
That recurring bug getting you down? Gonna get to it "someday". Hell, I know how to avoid it. I'll fix it next week. NO! DO IT NOW! If for any other reason, it will free up some RAM in your brain.

STOP WORSHIPING YOUR LANGUAGE
Hey... it's all assembly languange in the end. Is your clever C++ object really saving you time? Will you understand it later? Can you debug through it? Was it necessary? Did you spend 20 minutes trying to follow the inheritance? Keep it simple!

PROGRAMMING IS JUST A TOOL
Ok... you know how to use your saw and hammer... now build the damn house. Get some plans, cut the wood, and hammer it together. If the thing drives nails... then start pounding! Unless you are a hobbyist, then don't forget that the real product is the game - not the "programming itself." Granted, we are all geeks that love to polish our code. Nothing wrong with that until it interferes with the paycheck!

Nice topic indeed....

Learn more than one programming language

C and C++ are nice languages - and one can write almost anything using them - but do try to expand your mental view about the idea of software engineering. Different languages almost always try expose and fix weaknesses in common languages. I would recomend one to at least learn one functional language aside the common path of C++.

Assertions

It took me a very long time to realize how to best use these constructions. At first - when learned the concept of design by contract - I would assert anything which sounded like a contract (function->function). What is a bit troublesome with assertions is that - you assume that the programmer will fuck up. And surely we do - but not anything near the kind of stuff we try to protect ourselves by using assertions. 




FILE *openRead(const char *filename)

{

  assert(filename);

  FILE *fp = fopen(filename, "rb");

  if (!fp) throw runtime_error("could not open file");

  return fp;

}



I used to do this all the time. And a lot of people still do this - praising how assertions helped them trough the development. The basic idea is that you have to choose a path when you start developing, either you are keen on the fact that whenever your software crashes you know that there is a bug in your code - or else you want to create software that never crashes no matter what.

path 1: "Utilize assertion to the extent and quickly figure out whats wrong with your software when a assertion breaks"
path 2: "Monitor which codepaths could lead to a software termination, often (like in a graphics engine) the common path is where you load resources (textures/maps etc). Create default solutions for these errnous situations."

Well, I chose path 2 - but I still use assertions. Confusing? Maybe - but the fact that I only use assertions for catching logical failures - making sure my code behaves like I want it to - saves quite some time instead of having to step trough it once in a while.

...

Why do I find that the assertion in 'openRead' is not very useful. It catches me when I'm passing a invalid filename so what could possibly be bad about this construct? Well, software becomes very complex in short period of time. Trying to rewrite stuff is a safe way to fail (netscape). What you need do is to think that each function - each module - each program - is an deserted island. When you first implemented 'openRead' you new that you would only call it from your code - you could eliminate any bugs while debugging. 4 weeks later you add a simple script parser - which read filenames from a script and processes them. Well guess what - your function is being called with unverifyed data. Rewrite? No. Write code which verifies every external resource? No. Keep it simple....

...

Easy way of understanding hard bugs

Go to the toy store. Buy a large teddy bear. Place it near your computer. Whenever you think "why in gods name is the happening to me" - take a walk - visit the teddy bear. Explain the issues and your problems to him/her. 9/10 times you will automatically end the conversation with "Never mind, I figured out what's wrong".

...

Sorry about the ranting, but it nice to get stuff ventilated.

Perhaps some more book recommendations. These are for C, but a lot of it is valid for C++ too.

"C traps and pitfalls", Andrew Koenig.
A thin book, basically a catalog of language features that will bite you when you don't expect it. Was made before the previous C standard, but still good enough to have it.

"Expert C programming, Deep C Secrets", Peter van der Linden.
More introductory than expert. Similar to the previous book, thicker. Also has a C++ section, which is simply hilarious. Has a lot of funny stories (mostly about interesting mistakes made in the past).

"Safer C", Les Hatton.
Haven,'t finished this one yet, but looks interesting. For safety-critical programming with C. Games may not be in the same category as flight control software, but it may still be useful.

"The Practice of Programming", Brian W. Kernighan, Rob Pike.
Covers many languages. Good introduction to the subject, more accessible IMHO than , also shorter.
Those who are considering buying the Dutch version may be interested to know that the translation is questionable. Expecially the tendency to translate language keywords occasionally can be annoying.

Babar:

Good clarifications on Assert usage. Don't use Assert()s as a way of avoiding the graceful handling of a failure condition. Asserts() are best used to monitor the logic of your code. Although sometimes I use them as warnings to trap things that I know shouldn't happen, but can still be handled. I should probably add some flags to my Assert() macro.

Also, you said something that I would put on my 10 commandment list:

"What you need do is to think that each function - each module - each program - is an deserted island"

This is important. I call it "systems" programming (probably a misuse of the term.) Basically, I try to write the lower levels of my code the way somebody would write the operating system. Remove all assumptions of how it will be used, and concentrate on the internal task each routine will perform. Keep interfaces minimal and well defined. This is probably the hardest thing for beginners to learn because it takes experience and a breadth of understanding to >fully< appreciate this idea.

Again, know thou environment. Consider the following contrived example:

#include "stdafx.h"
#include
#include
#include

int main(int argc, char* argv[])
{
char *foo;

assert(foo);

if(foo)
{
printf("Should have asserted\n");
}

return 0;
}

Using MSVC the following will print "Should have asserted" (nevermind that the compiler did warn me about using an uninitialized variable). In debug builds MSVC inits local vars to 0xcccccccc which will never cause an assertion to fail. Instead MSVC offers the following functions:
IsBadCodePtr()
IsBadReadPtr()
IsBadStringPtr()
IsBadWritePtr()
(consult your online documentation for explanations)

Now adding the following assertion will catch what was intended:

#include "stdafx.h"
#include
#include
#include

int main(int argc, char* argv[])
{
const int MAX_LENGTH = 16;
char *foo;

assert(foo);

if(foo)
{
printf("Should have asserted\n");
}

assert(!IsBadStringPtr(foo, MAX_LENGTH));

printf("Assert should have fired!\n");

return 0;
}

In general, I think asserting on whether a element is NULL is generally useless since stack corruptions of variables or unitialization of variables will never trigger assertions.

Being my first time here, I'll write some short advices on some poor programming practices. I'm a C programmer not a C++ programmer, but what I'm going to speak about will also be of use to C++ programmers. I'm also not an english native speaker, so in advance I apologize any semantic, gramatical or otherwise lexical errors my post might contain.

One of the most normal type of errors, are overflows. They generally surpreend the programmer, when his code hasn't been enoughly stress-tested or otherwise designed security-wise/error-resistant.

For example, consider the case where you have defined a buffer that is 200 bytes in length. Your GUI is set up such that the user will add his name to an editbox, and that name will be copied using a function such as sprintf() or strcpy() to this buffer that you've previously defined. The problem arises when the name of the person is bigger than the actual buffer, although the programmer is conscious that there's no name bigger than 200 bytes, but does that avoid the error? the answer is no.

To avoid this errors, which are serious security breaches and can be exploited--which basicly means overwriting memory and execute arbitrary assembly code--the use should always avoid functions that can't limit its output by a size. Better functions to replace the forementioned ones would be snprintf() and strncpy() respectively. However, the latter can be as destructive as the previous one and the former even worse when improperly used--often happens as misconception from the programmer point of view.

Other situations where this type of bugs happen often is when using functions that accept format directives. These are: printf(), sprintf(), snprintf() and etc. Casual programmers are used to abuse this type of functions in a way that is similar to the following:

It looks pretty simple and safe. Now consider sending an %s or a %p or even an %n, and keep wondering what exactly hit your program that crashed it, certainly you know it now, but would you know it if a user give you a bug report about it? ;-) Also interesting to note is that this type of bugs, let's you examine the stack and may be used to overwrite protections or leak parts of memory that may contain risky information, like passwords, serial numbers, private keys, you-name-it. 




...

char buffer[200];



snprintf(buffer, sizeof(buffer) - 1, editbox->text);

MessageBox(. . . , buffer,  . . .); 

...



Integer and Unsigned-Integer conversion fsck ups.

Consider the simple and plain C program that accepts an unsigned int number and stores it on a signed integer. Consider the number that is actually stored is 0xffffffff, now consider it to be the value to allocate a string that is fed to malloc or some other sort of allocator, also that it will allocate space for a string, being a conscious programmer you'll add 1 for the leadding \0 byte. What exactly happens is that your 2^32-1 number turns out to be -1 + 1 which in the end becomes 0. When copying the string, you're sitted in your chair looking at the monitor and thinking "It's safe, because i've allocated enough space for the string, it will not crash" well, bad news, actually you requested an allocation of only 0 bytes, these are often more than just 0, but your used memory will become 0, does it will crash. This is often one of the types of bugs that are hard to detect and pose serious risk to the security and stability of your program. Also, integer arithmetic generally is notorious to make this type of errors to appear often, I think I gave the global view of the vulnerability and will not need to sum up anymore. 




int len = 0xffffffff;

char *cp;



cp = malloc(len + 1);

strcpy(cp, editbox->text);



It's also considered good programming practice to end all strings before passing them to a string-handling function with a zero-byte ('\0' or simply 0).

This type of errors I referenced in this post are mainly security issues, but occur often when you don't know how to program well or simply just don't think the program from a user point of view. Be careful with what date your program accepts, think of it as "What if i was able fool the logic of this program, how would I do so, what routes would I take for that ?!
